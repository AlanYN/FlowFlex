# test-verification：测试验证与覆盖率报告

## 职责定位

基于业务规范和技术方案生成完整测试用例，明确每个测试的具体步骤、预期结果和需要验证的问题，执行 Round-Trip 验证，确认交付物满足所有原始需求，输出需求覆盖率报告。

---

## 输入

| 来源 | 读取文件 | 关键字段 |
|------|---------|---------|
| requirements-analysis | `docs/{module-name}/requirements-analysis/context.md` | 用户故事摘要、AC 完整列表 |
| technical-design | `docs/{module-name}/technical-design/context.md` | 任务列表摘要、风险摘要、架构摘要 |

## 强制执行的 Skills（按序）

> **执行原则**：步骤 1 生成测试用例后，需用户确认覆盖范围；步骤 2 在代码实现后执行；步骤 3 生成最终报告。

### 第 1 步 — test-case-generator（测试用例生成）
读取 `.kiro/steering/skills/test-case-generator/SKILL.md`
目的：将所有 AC 和 RISK 转化为结构化测试用例，覆盖正向、异常、边界三类场景。

**测试框架选型（根据架构确认结果联动）**：

| 技术栈 | 后端测试框架 | 前端测试框架 |
|--------|------------|------------|
| C# + ASP.NET Core（默认） | xUnit + Moq + WebApplicationFactory | — |
| React 前端 | — | Vitest + React Testing Library + MSW |
| Vue 前端 | — | Vitest + Vue Test Utils + MSW |
| 新建项目（无已有框架） | 使用上表默认值 | 使用上表默认值 |
| 已有项目 | 扫描 `.csproj` / `package.json` 检测已有测试框架，优先沿用 | 同左 |

**覆盖要求**：
- 每条 AC 至少 1 条正向测试用例
- 每条含错误条件的 AC 额外 1 条异常测试用例
- 每条 🔴 Critical 风险至少 2 条测试用例（触发 + 边界）
- 每条 🟡 Medium 风险至少 1 条测试用例

**后端测试覆盖要求（C# 项目）**：
- 每个 API 端点至少 1 条集成测试（使用 `WebApplicationFactory`，验证完整请求/响应链路）
- 每个 Service 方法至少 1 条单元测试（使用 Moq mock Repository）
- 认证/权限端点必须包含：未认证请求返回 401、无权限请求返回 403
- 雪花 ID 端点必须验证：响应中 ID 字段类型为 string，不为 number

**前端测试覆盖要求（React / Vue 项目）**：
- 每个页面组件至少覆盖以下场景：
  - 初始加载状态（loading skeleton / spinner 是否渲染）
  - 数据正常返回后的渲染（列表条数、字段值是否正确显示）
  - 空数据状态（empty state 组件是否渲染）
  - 错误状态（API 失败时错误提示是否显示）
  - 用户交互（表单提交、按钮点击、分页切换等核心操作）
- 使用 MSW（Mock Service Worker）拦截 API 请求，不依赖真实后端
- 表单组件必须测试：必填校验、格式校验、提交成功/失败反馈
- 列表页必须测试：分页、搜索/筛选条件变化后重新请求

**每个测试用例必须包含**：
- 测试目的（验证什么问题）
- 前置条件（系统状态、数据准备）
- 具体操作步骤（原子级，可直接执行）
- 预期结果（明确可验证，不得使用"正常显示"等模糊描述）
- 测试数据示例（边界值、异常值）

输出：完整测试用例列表（TC-001, TC-002...）→ 写入 `specs/.../test-verification/design.md`

---

> ⏸️ **确认节点**：步骤 1 完成后，展示测试用例总览和覆盖矩阵，等待用户确认覆盖范围无遗漏后，才能标记阶段完成。

---

### 第 2 步 — auto-test-runner（自动化测试执行）
读取 `.kiro/steering/skills/auto-test-runner/SKILL.md`
目的：检测项目测试框架，尝试执行自动化测试，收集真实结果。

**重要**：若项目代码尚未实现（无可运行的测试文件），所有 TC 状态标记为 `pending`，不得伪造执行结果。仅在真实测试运行后才更新为 `pass` / `fail` / `skip`。

输出：执行结果表格 + 覆盖率报告 → 写入 `specs/.../test-verification/design.md`

### 第 3 步 — 生成测试问题清单（ISSUE LIST）
基于测试用例，生成开发阶段需要重点关注的问题清单：

**问题分类**：
- 🔴 **安全问题**：认证绕过、SQL 注入、XSS、CSRF、越权访问
- 🟡 **数据问题**：边界值处理、空值处理、并发写入、数据一致性
- 🟢 **体验问题**：加载状态、错误提示、空状态、响应式布局
- 🔵 **性能问题**：大数据量分页、慢查询、内存泄漏

每个问题必须关联到具体的 TC 编号。

输出：测试问题清单 → 写入 `specs/.../test-verification/design.md` 问题清单部分

## Round-Trip 验证规范

### 验证步骤

1. 读取 `raw-requirements.md`，列出所有原始需求
2. 确认每条原始需求都有对应的 AC
3. 确认每条 AC 都有对应的 TC
4. 确认每条 TC 都有执行结果（pass / fail / skip / pending）
5. 统计覆盖率：有对应 TC 的需求数 / 总需求数

### 验证矩阵

```markdown
| 原始需求 | 关联 AC | 关联 TC | 执行结果 | 覆盖状态 |
|---------|---------|---------|---------|---------|
| {需求描述} | AC-001-1 | TC-001 | pass | ✅ passed |
| {需求描述} | AC-002-1 | TC-002 | pending | ⏳ pending |
| {需求描述} | - | - | - | ⚠️ not_covered |
```

---

## 执行规范

### TC 状态规则

| 状态 | 含义 |
|------|------|
| `pending` | 测试用例已定义，但代码未实现，无法执行 |
| `pass` | 测试运行并通过 |
| `fail` | 测试运行但断言失败，必须填写 failureReason |
| `skip` | 测试被跳过（手动测试或环境限制） |

**严禁**在代码未实现时将状态标记为 `pass`。

### 完成标准

- 执行全部 2 个 Skill（按顺序）
- 所有 TC 均有执行状态（`pending` 是合法状态，代表等待代码实现）
- 覆盖率报告已生成，所有需求均有对应状态
- 标记 test-verification 为 `completed`
