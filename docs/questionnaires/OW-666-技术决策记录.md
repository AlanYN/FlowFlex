# OW-666: Repeatable Section — 技术决策记录

本文档记录 OW-666 实现过程中经过讨论确认的技术决策，作为开发执行的依据。

---

## 1. 页面导航

**决策：一个 Repeatable Section = 一个页面，所有 Group 在同一页垂直排列。**

- 不把每个 Group 拆成独立页面
- Progress dots 数量不随 Group 增减变化
- 页面内容多时靠滚动浏览

---

## 2. 前端数据结构

**决策：Repeatable Section 的表单数据使用嵌套结构 `sectionGroups[sectionId][groupIndex][questionId]`。**

```typescript
// 非 Repeatable —— 保持现有 formData
formData["q1"] = "Acme"

// Repeatable —— 独立结构
sectionGroups["s2"][0]["q10"] = "LA Warehouse"
sectionGroups["s2"][0]["q11"] = "Alice"
sectionGroups["s2"][1]["q10"] = "NJ Warehouse"
sectionGroups["s2"][1]["q11"] = "Eve"
```

选择理由：删除组时只需 `splice(groupIndex, 1)`，无需重建 key；校验时逐组遍历天然支持；不污染现有 formData。

---

## 3. 完成率计算

**决策：按实际组数计入 `totalQuestions`。**

- 公式：`answeredQuestions / (普通题数 + Repeatable题数 × 实际组数)`
- 用户加空组 → 进度率下降；删组 → 进度率上升
- 与 Required 校验逻辑一致

---

## 4. jumpRules（条件跳转）

**决策：Repeatable Section 内的 Question 禁止配置 jumpRule。**

- Admin 端：属于 Repeatable Section 的 Question，jumpRule 配置入口隐藏
- Assignee 端：`getJumpTargetSection()` 忽略来自 Repeatable Section 的答案
- 原因：多组同一 Question 会产生矛盾的跳转指令，无法解决

---

## 5. Draft 回填

**决策：渲染最新模板，旧答案按 questionId 匹配回填，无法匹配的留空。**

- 从 `sectionInstances` 读取已存的组数据，动态创建对应数量的 Group
- 模板中新增的 Question 在所有 Group 中留空
- 不做模板快照

---

## 6. 后端校验扩展

**决策：`ValidateNumberAnswers` 扩展覆盖 `sectionInstances[].responses[]`。**

- 在现有遍历 `responses[]` 的基础上，增加一层遍历 `sectionInstances`
- 改动量小，防止 Repeatable 数据绕过校验

---

## 7. 导出格式

**决策：每组每题一行，新增 "Group" 标识列。**

导出效果：
```
| Section         | Group   | No. | Question        | Answer       |
|-----------------|---------|-----|-----------------|--------------|
| Company Info    |         | 1   | Company name?   | Acme         |
| Facility Config | Group 1 | 1   | Facility Name   | LA Warehouse |
| Facility Config | Group 1 | 2   | Account Manager | Alice        |
| Facility Config | Group 2 | 1   | Facility Name   | NJ Warehouse |
| Facility Config | Group 2 | 2   | Account Manager | Eve          |
```

非 Repeatable Section 的 Group 列留空。

---

## 8. 模板渲染策略

**决策：v-if / v-else 两份模板，用 `getTarget(groupIndex)` 函数代理数据源。**

```typescript
function getTarget(groupIndex?: number) {
    if (currentSection.value.isRepeatable && groupIndex !== undefined) {
        return sectionGroups[currentSection.value.id][groupIndex];
    }
    return formData;
}
```

- 非 Repeatable 模板保持现有代码不动
- Repeatable 模板在外层套 Group 循环，内部 `v-model` 绑定到 `getTarget(groupIndex)[question.id]`
- 不抽子组件，对旧逻辑影响最小

---

## 9. sectionInstances 内部 response 结构

**决策：与顶层 `responses[]` 完全一致。**

```json
{
    "questionId": "...",
    "question": "问题文本",
    "answer": "值",
    "type": "short_answer",
    "responseText": "...",
    "lastModifiedBy": "...",
    "lastModifiedAt": "..."
}
```

便于后端复用同一套遍历/校验逻辑。

---

## 10. 删除 Group 确认交互

**决策：有数据的 Group 弹确认框，空 Group 直接删除。**

- 检查逻辑：`Object.values(group).some(v => v != null && v !== '')`
- 有值时弹 `ElMessageBox.confirm("确定要删除 Group N 吗？该组已填写的数据将丢失")`
- 至少保留 1 组（最后一组隐藏删除按钮）

---

## 11. Preview 模式

**决策：Preview 保持可交互，支持增删 Group 和填写。**

- 和 Assignee 端复用同一套渲染逻辑
- 数据不保存
- Admin 可以预览"加了多组是什么效果"

---

## 12. 问题编号

**决策：每组内独立编号（1.1, 1.2, 1.3），靠 Group 标题区分。**

- Group 1 和 Group 2 的同一个 Question 编号相同
- 不跨组连续编号，不使用三级编号
- 增删组不影响编号稳定性

---

## 13. Required 校验定位

**决策：跳转到第一个有错误的 Section 页 + 滚动到具体 Group 的具体 Question。**

- 遍历校验结果，找到第一个 error 的 `{ sectionIndex, groupIndex, questionId }`
- 自动切换 `currentSectionIndex` + `scrollIntoView`
- 用户修复后再次提交定位到下一个错误

---

## 14. 模板变更对 Draft 的影响

**决策：Submitted 不变，Draft 跟新模板走。**

- 普通 → Repeatable：旧 `responses[]` 里属于该 Section 的答案丢失，初始化 1 组空白 Group
- Repeatable → 普通：`sectionInstances` 数据不渲染，等同丢失
- 接受丢失，不做自动迁移

---

## 15. 组数上限

**决策：前端硬上限 50 组。**

- 达到 50 组时隐藏 "Add Another" 按钮
- 不需要后端参与
- 后续做 Min/Max 时可改为 Admin 可配置

---

## 16. Group 滚动位置记忆

**决策：不做。**

- 不记住用户在 Repeatable Section 里滚动到哪个 Group
- MVP 不增加 `currentGroupIndex` 字段
- 多数场景 3-5 组，滚动成本低

---

## 17. 变更追踪

**决策：MVP 不追踪 `sectionInstances` 的变更历史。**

- `ProcessAnswerChangesAsync` 保持原样，只处理 `responses[]`
- Repeatable 部分无 `changeHistory`
- 原因：组增删 + index 重排导致对比逻辑复杂，投入产出比不高

---

## 18. 孤儿数据清理

**决策：提交时只收集当前模板中存在的 Question（自然清理）。**

- `transformFormDataForAPI` 按当前模板的 `section.questions` 遍历收集
- 已从模板中删除的 Question 不会被收集进 `sectionInstances`
- 不需要额外写清理代码，是遍历逻辑的自然结果
