## 二、字段级别差异记录机制（Old Value / New Value）

### 2.1 存储方式

**没有独立的 `old_value` / `new_value` 列。** 采用的是 JSON 快照对比模式：

| 字段                     | 存储内容                                                             | 说明               |
| ------------------------ | -------------------------------------------------------------------- | ------------------ |
| `before_data` (JSONB)    | `{"CaseName":"ABC Inc","Priority":"Medium","Ownership":123,...}`     | 修改前整个对象快照 |
| `after_data` (JSONB)     | `{"CaseName":"ABC Logistics","Priority":"High","Ownership":456,...}` | 修改后整个对象快照 |
| `changed_fields` (JSONB) | `["CaseName","Priority","Ownership"]`                                | 哪些字段发生了变化 |

### 2.2 后端记录流程（以 Case 更新为例）

```csharp
// OnboardingCrudService.cs - ProcessPostUpdateAsync 方法

// 1. 修改前先快照整个对象
var originalValues = CaptureOriginalValues(entity);  // 返回 Tuple，逐字段保存原值
var beforeData = SerializeBeforeData(entity, beforeWorkflowName);
// → {"CaseName":"ABC Inc", "Priority":"Medium", "Ownership":123, "ViewTeams":"[...]", ...}

// 2. 执行修改
_mapper.Map(input, entity);
await SafeUpdateOnboardingAsync(entity);

// 3. 序列化修改后数据（格式与 beforeData 一致）
var afterData = JsonSerializer.Serialize(new { CaseName = entity.CaseName, Priority = entity.Priority, ... });

// 4. 逐字段比较，记录哪些字段变了
var changedFields = new List<string>();
if (originalValues.CaseName != entity.CaseName) changedFields.Add("CaseName");
if (originalValues.Priority != entity.Priority) changedFields.Add("Priority");
if (originalValues.Ownership != entity.Ownership) changedFields.Add("Ownership");
if (originalValues.ViewPermissionMode != (int)entity.ViewPermissionMode) changedFields.Add("ViewPermissionMode");
if (originalValues.ViewTeams != entity.ViewTeams) changedFields.Add("ViewTeams");
if (originalValues.ViewUsers != entity.ViewUsers) changedFields.Add("ViewUsers");
// ... 共 14 个字段逐一比较

// 5. 写入 ChangeLog（三者同时记录）
await _onboardingLogService.LogOnboardingUpdateAsync(
    entity.Id,
    entity.CaseName,
    beforeData: beforeData,       // 修改前 JSON
    afterData: afterData,         // 修改后 JSON
    changedFields: changedFields   // 变化的字段名列表
);
```

### 2.4 前端解析方式

```typescript
// apis/ow/change-log.ts - parseStaticFieldChanges 函数
// 从 before_data / after_data JSON 中提取每个字段的差异值
export function parseStaticFieldChanges(
  beforeData,
  afterData,
  changedFields,
  userMap,
) {
  const before = JSON.parse(beforeData);
  const after = JSON.parse(afterData);
  const changes = [];

  // 逐字段提取 Old/New
  changedFields.forEach((field) => {
    changes.push({
      fieldName: field, // 如 "CaseName"
      beforeValue: before[field], // 如 "ABC Inc"
      afterValue: after[field], // 如 "ABC Logistics"
    });
  });
  return changes;
}
```

### 2.5 与需求对比

| 需求要求  | 现有实现                          | 差异                      |
| --------- | --------------------------------- | ------------------------- |
| Timestamp | `operation_time` 字段             | ✅ 完全满足               |
| User      | `operator_name` 字段              | ✅ 完全满足               |
| Field     | `changed_fields` JSON 数组        | ✅ 有，但存的是字段名列表 |
| Old Value | `before_data` JSON 中对应字段的值 | ✅ 数据在，需解析提取     |
| New Value | `after_data` JSON 中对应字段的值  | ✅ 数据在，需解析提取     |

> **结论**：需求的 Timestamp / User / Field / Old Value / New Value 五要素全部已有数据支撑，只是不是"每字段一行"的扁平格式，而是"整对象快照 + 变化字段列表"的组合格式。如需推送给外部系统，需要在推送时从 JSON 中拆解出逐字段 diff。

## 三、日志触发方式

### 3.1 结论：直接嵌入式调用（Inline Call）

**不是中介者模式、不是 AOP、不是事件驱动。**  
所有 ChangeLog 写入都是业务 Service 显式注入 Log Service，在业务完成后手动调用。

### 3.2 两种调用模式

**模式 1（主流）：通过 BackgroundTaskQueue fire-and-forget**

```csharp
// OnboardingStatusService.cs
var result = await _onboardingRepository.UpdateStatusAsync(id, "Paused");
if (result)
{
    _backgroundTaskQueue.QueueBackgroundWorkItem(async token =>
    {
        try
        {
            await _onboardingLogService.LogOnboardingPauseAsync(id, entity.CaseName, reason: null);
        }
        catch (Exception ex)
        {
            _logger.LogError(ex, "Failed to log...");
        }
    });
}
```

**模式 2（少数）：直接 await（包在 try-catch 不阻塞主流程）**

```csharp
// ChecklistTaskCompletionService.cs
await _operationChangeLogService.LogOperationAsync(
    operationType: operationType,
    businessModule: BusinessModuleEnum.ChecklistTask,
    businessId: task.Id,
    onboardingId: onboarding.Id,
    stageId: currentStage?.Id ?? 0,
    operationTitle: $"Task Completed: {task.Name}",
    operationDescription: operationDescription,
    beforeData: JsonSerializer.Serialize(beforeData),
    afterData: JsonSerializer.Serialize(afterData),
    changedFields: changedFields,
    extendedData: JsonSerializer.Serialize(extendedData)
);
```

## 四、BackgroundTaskQueue 机制

**接口**: `Infrastructure/Services/IBackgroundTaskQueue.cs`

```csharp
public interface IBackgroundTaskQueue
{
    void QueueBackgroundWorkItem(Func<CancellationToken, Task> workItem);
    Task<Func<CancellationToken, Task>> DequeueAsync(CancellationToken cancellationToken);
}
```

**实现**: `BackgroundTaskQueue`（Singleton, ConcurrentQueue + SemaphoreSlim）  
**消费者**: `BackgroundTaskService`（HostedService，循环 Dequeue 执行）  
**注册**: `Program.cs`

```csharp
builder.Services.AddSingleton<IBackgroundTaskQueue, BackgroundTaskQueue>();
builder.Services.AddHostedService<BackgroundTaskService>();
```

**特性**：

- 内存队列
- 进程重启会丢失未执行任务
- 无重试机制
- 无持久化
- 适合"允许丢失"的异步操作（如日志写入）
