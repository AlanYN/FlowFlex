# ADR 格式

ADR 存放在 `docs/adr/` 中，使用顺序编号：`0001-slug.md`、`0002-slug.md` 等。

延迟创建 `docs/adr/` 目录 — 只在需要第一个 ADR 时才创建。

## 模板

```md
# {决策的简短标题}

{1-3 句话：上下文是什么，我们决定了什么，以及为什么。}
```

就这样。一个 ADR 可以只是一个段落。价值在于记录*做出了*决策以及*为什么* — 而不是填写各个章节。

## 可选章节

只在它们确实增加价值时才包含。大多数 ADR 不需要它们。

- **Status** frontmatter（`proposed | accepted | deprecated | superseded by ADR-NNNN`）— 在决策被重新审视时有用
- **Considered Options** — 只在被拒绝的替代方案值得记住时
- **Consequences** — 只在非显而易见的下游影响需要被指出时

## 编号

扫描 `docs/adr/` 找到最高的现有编号并加一。

## 何时提供 ADR

以下三个条件必须全部满足：

1. **难以逆转** — 以后改变主意的成本是有意义的
2. **没有上下文会令人惊讶** — 未来的读者会看着代码想"他们到底为什么这样做？"
3. **真正权衡的结果** — 有真正的替代方案，你因为特定原因选择了一个

如果决策容易逆转，跳过它 — 你直接逆转就好。如果它不令人惊讶，没人会想为什么。如果没有真正的替代方案，除了"我们做了显而易见的事"之外没什么好记录的。

### 什么符合条件

- **架构形状。** "我们使用 monorepo。" "写模型是事件溯源的，读模型投影到 Postgres。"
- **上下文之间的集成模式。** "Ordering 和 Billing 通过领域事件通信，而不是同步 HTTP。"
- **带有锁定效应的技术选择。** 数据库、消息总线、认证提供商、部署目标。不是每个库 — 只是那些需要一个季度才能替换的。
- **边界和范围决策。** "Customer 数据由 Customer 上下文拥有；其他上下文仅通过 ID 引用。" 明确的"不"和"是"一样有价值。
- **对显而易见路径的刻意偏离。** "我们使用手写 SQL 而不是 ORM，因为 X。" 任何合理读者会假设相反做法的地方。这阻止下一个工程师"修复"一个刻意的选择。
- **代码中不可见的约束。** "由于合规要求我们不能使用 AWS。" "由于合作伙伴 API 合约，响应时间必须低于 200ms。"
- **当拒绝不明显时的被拒替代方案。** 如果你考虑了 GraphQL 并因为微妙的原因选择了 REST，记录下来 — 否则六个月后有人会再次建议 GraphQL。
